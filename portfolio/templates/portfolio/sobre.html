1. Arquitetura MVT

O Django segue o padrão arquitetural MVT (Model-View-Template), que separa a aplicação em três camadas com responsabilidades distintas:

  • Model (models.py) — Define a estrutura dos dados (tabelas da BD).
  • View (views.py) — Contém a lógica da aplicação. Recebe pedidos, interage com Models, e devolve respostas.
  • Template (.html) — Define como os dados são apresentados ao utilizador.

Desenho da Arquitetura

Fluxo de Informação

  1. O utilizador clica num link no browser.
  2. O browser envia um pedido HTTP (ex: GET /portfolio/projetos/).
  3. O urls.py mapeia o URL para uma view.
  4. A view chama métodos do Model para obter dados (ex: Projeto.objects.all()).
  5. O Model consulta a base de dados e devolve os dados à view.
  6. A view passa os dados a um template através do render().
  7. O template renderiza HTML formatado com os dados.
  8. O browser recebe o HTML e mostra a página ao utilizador.

2. Modelação

A modelação dos dados foi pensada para representar o meu percurso académico, profissional e técnico, com várias relações entre entidades.

Diagrama Entidade-Relação

Entidades Principais

  • Licenciatura — informação sobre o curso (LEI).
  • UnidadeCurricular — UCs da licenciatura, lecionadas por docentes.
  • Docente — professores que lecionam UCs.
  • Projeto — projetos académicos, associados a uma UC.
  • Tecnologia — tecnologias usadas (com tipo: Frontend, Backend, etc.).
  • Tipo — categoria das tecnologias.
  • Competencia — competências técnicas e soft skills.
  • Formacao — formações académicas e externas.
  • TFC — Trabalhos Finais de Curso.
  • ExperienciaProfissional — experiência profissional.
  • MakingOf — apontamentos do desenvolvimento.

Principais Relações

  • Licenciatura tem várias UnidadesCurriculares (1:N)
  • UnidadeCurricular é lecionada por Docentes (M:N)
  • UnidadeCurricular origina Projetos (1:N)
  • Projeto usa Tecnologias (M:N)
  • Projeto trabalha Competências (M:N)
  • Tecnologia pertence a um Tipo (N:1)

5. Repositório GitHub

Todo o código está versionado no GitHub. Cada commit representa uma iteração do desenvolvimento, documentada no Making-Of.

📦 Ver Repositório no GitHub →

Como uso o Git/GitHub

  • Commits frequentes com mensagens descritivas seguindo o padrão feat:, fix:, etc.
  • Branch main sempre estável.
  • Push para o remoto após cada iteração testada.

Vantagens

  • Histórico completo de alterações.
  • Possibilidade de reverter alterações se algo correr mal.
  • Backup automático na cloud.
  • Trabalho desde qualquer máquina (com Codespaces).
